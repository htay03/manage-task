# 進捗管理アプリ 開発メモ（発表用・PPT下書き）

> 目的：チームの進捗を管理し、進捗サマリを**定期メールで自動共有**するアプリ。
> このメモは発表資料（PPT）を作るための素材。各見出しがスライド1枚のイメージ。

---

## 0. 概要 / 技術スタック

- **フロント／サーバー**：Next.js（App Router）＋ TypeScript ＋ Tailwind CSS
- **バックエンド**：Supabase（認証 / DB(PostgreSQL) / RLS / Cron）
- **メール送信**：GMO の SMTP（`smtp12.gmoserver.jp`）＋ nodemailer
- **ホスティング**：Vercel（東京リージョン）
- **チーム開発**：GitHub（`main` は常に動く完成版、機能ごとに `feature/*` ブランチ → PR → マージ）

---

## 1. 実装した機能（実装した順）

### ① 認証・トップ画面（土台）
- **仕組み**：Supabase Auth（メール＋パスワード）。
  - `AuthProvider`（ログインセッションを監視）＋ `AuthGate`（未ログインは `/login` へ、ログイン済みは `/` へ誘導）で画面をガード。
  - ログイン / 新規登録 / ログアウト。
- **💡 詰まった**：本番で新規登録すると特定のメールアドレスで確認メールが届かない。
  → Supabase標準メールの制限＋別ドメイン間送信で迷惑メール判定されやすい、が原因。

### ② タスク登録・一覧
- **仕組み**：Supabase に `tasks` テーブル（`title/assignee/due_date/status/parent_id/…`）。
  - RLS（Row Level Security）で「ログイン済みユーザーは全タスクを操作可」＝チーム共有。
  - `src/lib/tasks.ts`（`listTasks`／`createTask`）＋ `/tasks` の登録フォーム＆一覧。

### ③ タスクの親子関係
- **仕組み**：`parent_id`（自己参照）で親タスク配下に子タスク（2階層）。
  - 一覧で子タスクを親の下にインデント表示。フォームで「親タスク」を選択。

### ④ 状態の更新
- **仕組み**：一覧の各タスクのドロップダウン（未着手／進行中／完了）を変えると即 Supabase 更新（`updateTaskStatus`）。楽観的更新（画面を先に反映、失敗したら戻す）。

### ⑤ チーム全体の進捗率
- **仕組み**：`taskProgress()` が「完了数 ÷ 全体数」を計算。`/tasks` 上部とトップ画面「今日のようす」に表示（＝一箇所の計算を両方で使い回し）。

### ⑥ メール共有の設定画面
- **仕組み**：`email_settings` テーブル（送信先／頻度／曜日／送信時刻／有効無効）。
  - `/tasks/mail` で設定を保存（チーム共通の1設定を upsert）。

### ⑦ 進捗サマリのメール送信＋**自動送信**（必須要件・最重要）
- 「設定した時刻に、進捗サマリを**自動で**メール送信する」＝進捗管理の必須要件の最後のピース。
- ボリュームが大きいので、**仕組みと詰まった点は第2章で詳しく**（4段階＋Y2-b／Y2-c）。
  ＝ この機能がこのプロジェクトの“山場”。

## 2. メール自動送信（最重要・詳しく）

「設定した時刻に、進捗サマリを自動でメール送信する」を、**4段階**で作った。

### 段階1：メール本文を作って送る処理
- **どこで動くか**：Next.js の **API ルート `/api/send-summary`**（**サーバー側** / Node ランタイム）。ブラウザではなくサーバーで実行する（SMTP接続やサービスロールキーなど、外に出せないものを扱うため）。
- **本文生成**：`buildProgressSummary()` が件名・本文（テキスト版＋HTML版）を組み立てる。
  - チーム全体の進捗 / 状態別件数 / 期限アラート / 前回からの差分 / 担当者別。
- **送信処理**：`nodemailer`（`src/lib/mailer.ts`）で SMTP に接続してメール送信。
- **DB読み取り**：サーバーからRLSを越えて「全タスク」と「送信設定」を読むため、**サービスロールキー**（`supabaseAdmin`）を使用。
- **エンドポイントの保護**：誰でも叩けないよう、**ログイン中ユーザーのトークン**か **`CRON_SECRET`** を持つ呼び出しだけ許可。
- **確認の入口**：メール設定画面 `/tasks/mail` に **「テスト送信」ボタン**を用意（自分のアドレス宛に1通送って確認できる）。
- **この段階の結果**：**ローカル（自分のPC）からのテスト送信は成功し、メールも実際に届いた** ✅
  - ⚠️ ただしこの「成功」が後の落とし穴。**自分のPC＝日本のIPだから通っていただけ**で、クラウドから送ると弾かれる（→ 段階3）。
- **💡 小さな詰まり**：最初のローカルテストで `535 authentication failed`。
  - 原因は SMTPパスワードに含まれる `#`。`.env.local` では `#` 以降がコメント扱いになり、パスワードが途中で切れていた → **値をダブルクォートで囲んで**解決。

### 段階2：認証、GMOへの接続（＋SMTPシークレット5つ）
- GMO の SMTP サーバー（`smtp12.gmoserver.jp`）に **SMTP認証してメール送信**。
- **秘密情報はコードに直書きしない**。環境変数／シークレットで管理：
  - **SMTP情報5つ**を登録 → `SMTP_HOST` / `SMTP_PORT` / `SMTP_USER` / `SMTP_FROM` / `SMTP_PASSWORD`
  - 最初は **Supabase の Edge Function Secrets** に5つ登録（Edge Function方式のとき）。
  - 後に方式変更に伴い **Vercel の環境変数**にも同じ5つ＋サービスロールキー＋`CRON_SECRET` を登録。
  - `.env.local`（Git管理外）はローカル用。`NEXT_PUBLIC_` は付けない（ブラウザに出さない＝サーバー専用）。
- **💡 詰まった**：
  - パスワードに `#` が含まれ、`.env` で `#` 以降がコメント扱いになり認証失敗 → **ダブルクォートで囲んで**解決。
  - Supabase のシークレット登録に **admin 権限が必要**で最初は登録できず（オーナー対応で解決）。
  - `535 authentication failed`：シークレットの値に引用符や空白が混入していて失敗 → 入れ直しで解決。

### 段階3：日本リージョンから送信ができる（最大の山）
- 当初は **Supabase Edge Function（Deno＋denomailer）** で送ろうとした。
- **💡 詰まった（本命）**：GMO が **国外IPからの送信を拒否**（`554 Incorrect country code SG`）。
  - Edge Function は **シンガポール（ap-southeast-1）** で動くため弾かれた。
  - ローカルテストが成功していたのは「自分のPC（日本）から送っていたから」と判明。
- **解決（具体的に何をしたか）**：送信を **Vercel の東京リージョン（hnd1）** で動かし、日本のIPから出るようにした。実際の手順：
  1. `vercel.json` に `{"regions": ["hnd1"]}` を追加（サーバー関数を**東京リージョンに固定**）。
  2. `/api/send-summary` に **`CRON_SECRET` での認証**を追加（後のスケジューラや `curl` が、ログイン無しで叩けるように）。
  3. リポジトリを **公開（public）化**（オーナー対応）→ 自分のアカウントに **フォーク**（`htay03/manage-task`）。
  4. フォークを **自分の個人Vercel** にインポート。
  5. Vercelの **環境変数** に「SMTP 5つ ＋ サービスロールキー ＋ `CRON_SECRET`」を登録（Production/Preview/Development すべて）。
  6. 送信ルート入りのブランチを **フォークの `main` に流し込んで** デプロイ。
  7. `curl` でエンドポイントを叩いてテスト → **メール受信（GMO通過）を確認** ✅
- **途中でさらに詰まった**：
  - （最初のEdge Function方式のとき）`denomailer` でポート587（STARTTLS）が `invalid cmd` → **ポート465（SSL）に変更**／`535` 認証失敗（シークレット値のズレ）。
  - Vercelの**デプロイ個別URL**が **Deployment Protection** で 401（"Redirecting…"）→ **本番URLを使用**／**Vercel Authentication を無効化**（エンドポイントは自前 `CRON_SECRET` で防御）。
  - 組織リポジトリを個人Vercelに繋げない（権限）→ **公開化＋フォーク**（詳細は下）。

#### なぜ「フォーク」する必要があったのか（詳しく）
- **やりたかったこと**：送信処理を **日本（Vercel東京リージョン）** で動かす。そのため、送信コードを**自分がコントロールできるVercelプロジェクト**にデプロイする必要があった。
- **ぶつかった壁は2つ**：
  1. **チームの本番Vercel（オーナーのチーム）** … 環境変数やリージョン設定を変えるには **オーナー権限**が必要で、自分では触れない。
  2. **自分の個人Vercelに組織リポジトリ（`share-re/manage-task`）を繋ごうにも一覧に出ない** … 組織リポジトリをVercelに追加するには **組織オーナーの承認**が必要（また権限の壁）。
- **解決＝フォーク**：リポジトリを **公開（public）** にしたので、**自分のアカウントに複製（フォーク）** して `htay03/manage-task` を作成。
  - フォークは**自分の個人リポジトリ**なので、**個人Vercelに自由に接続**でき、**環境変数・東京リージョン・デプロイを自分だけで設定**できる。
  - つまりフォーク＝**「送信処理を日本リージョンで動かすための、自分専用のデプロイ置き場（ミラー）」**。
  - 開発の本流は `share-re` のまま。フォークには最新コードを流し込み（push）、Vercel がそれをデプロイ。
- **要点**：**オーナーに依頼せず、自力で“東京リージョンのデプロイ環境”を用意する**ためにフォークを使った。
  - 補足：フォーク作成時に「main だけコピー」になっていて対象ブランチが無い、という小さな詰まりもあった（対象ブランチをフォークの `main` に流し込んで解決）。

### 段階4：スケジューラで自動化
- `/api/send-summary` に **スケジュール判定** を追加。
  - `{scheduled:true}` で呼ばれたら「今が設定時刻か（**日本時間で判定**）／有効か／今日まだ送っていないか」を見て、該当時だけ送信。
  - `last_sent_at` 列で **二重送信を防止**（送ったら記録、同じ日は再送しない）。
  - 手動／テスト送信は今まで通り即送信。

#### Y2-b：スケジュール判定の「手動テスト」（自動化前の事前確認）
- 目的：Cron を組む前に「今が送る時刻か？」の判定ロジックが正しいか手で1回確認。
- **skipテスト**：送信時刻を「未来」にして `{scheduled:true}` → `not due`（送らない）。
- **sendテスト**：送信時刻を「今の数分前」＋有効ON → 送信され `last_sent_at` に記録。
- テスト後は **送信時刻を戻し、`last_sent_at` を null に**リセット。
- **実行コマンド（手動でエンドポイントを叩く）**：cmd で1行。
  ```
  curl -i -X POST "https://<Vercel本番URL>/api/send-summary" -H "Authorization: Bearer <CRON_SECRET>" -H "Content-Type: application/json" -d "{\"scheduled\":true}"
  ```
  - `{"scheduled":true}` あり → スケジュール判定を通す（時刻でないと `not due`）。
  - `{"scheduled":true}` なし → 判定を無視して**即送信**（通常のテスト送信）。

#### Y2-c：Supabase Cron 登録（＝自動化の完成）
- Supabase の **`pg_cron` ＋ `pg_net`** 拡張を使用。
- **15分ごと**（`*/15 * * * *`）に `net.http_post` で Vercel の `/api/send-summary` を叩く。
  - ヘッダに `Authorization: Bearer <CRON_SECRET>`、ボディに `{"scheduled": true}`。
- 動き：Cronが15分ごとに起動 → 設定時刻になったら送信 → その日は重複させない。
- **登録SQL（Supabase SQL Editor）**：
  ```sql
  create extension if not exists pg_cron;
  create extension if not exists pg_net;

  select cron.schedule(
    'send-summary-15min',
    '*/15 * * * *',
    $$
    select net.http_post(
      url := 'https://<Vercel本番URL>/api/send-summary',
      headers := jsonb_build_object(
        'Content-Type', 'application/json',
        'Authorization', 'Bearer <CRON_SECRET>'
      ),
      body := jsonb_build_object('scheduled', true)
    );
    $$
  );
  ```
  - `<Vercel本番URL>` は自分のフォークVercelのURL、`<CRON_SECRET>` は Vercel環境変数に入れた合鍵（発表資料には実値を載せない）。

### メール本文の中身（拡張テンプレート）
1. **チーム全体の進捗**（完了X / 全Y）
2. **要注意**：期限超過（◯日超過）／本日締切／**あと◯日**（3日以内）
3. **前回からの変化**：完了になった／新しく追加（`last_sent_at` 基準）
4. **担当者別の進捗**

## 3. 詰まったこと（横断まとめ）※発表の“山場”になりやすい

- **権限の壁**：ブランチ保護 / Vercel連携 / Supabaseシークレット / 組織リポジトリのVercel接続 …
  → いずれも **組織オーナーの権限**が必要で、都度依頼 or 回避策が要った。
- **GMOの海外制限**：クラウド（シンガポール）から送れず → **Vercel東京リージョン**で解決。
- **SMTPパスワードの `#`／引用符**：環境変数の扱いでハマった。
- **Vercelのデプロイ保護**：エンドポイントが叩けず → 本番URL／保護OFF＋自前トークンで防御。
- **組織リポジトリを自力デプロイ**：**公開化＋フォーク**して個人Vercelへ。

## 4. 発展（追加で実装）

- **パスキー（WebAuthn）認証**をメール＋パスワードと**併用**で追加。
  - `signInWithPasskey()`（ログイン）／`registerPasskey()`（登録）。
  - 注意：パスキーは**ドメイン（`rp_id`）に固定**。localhost用と本番用で設定が別。プレビューのように毎回URLが変わる環境では使えない → **安定ドメイン**で使う。
